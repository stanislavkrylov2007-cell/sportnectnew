# Нефункциональные требования

| Категория | Требование | Приоритет | Обоснование |
|-----------|-------------|-----------|-------------|
| **Безопасность** | Пароли хранятся в виде хеша. После 5 неудачных попыток входа — блокировка 5 мин. Передача данных — HTTPS. | MUST HAVE | Защита от утечек и перехвата. Обязательно для приложений с персональными данными. |
| **Производительность** | Загрузка главной страницы < 2 сек | MUST HAVE | Медленный интерфейс отпугивает пользователей. Критично для удержания. |
| **Надёжность** | Ежедневное резервное копирование БД, Uptime ≥ 99.5% | MUST HAVE | Доступность в любое время важна для спонтанных игр. Резервные копии — защита от сбоев. |
| **Удобство** | Регистрация и создание профиля ≤ 3 мин. | SHOULD HAVE | Низкий порог входа для новичков. |
| **Совместимость** | Поддержка Chrome, Firefox, Safari, Яндекс.Браузер. Интеграция с Яндекс.Картами / Google Maps | MUST HAVE | Без карт приложение не работает. Охват браузеров — >95% аудитории. |
| **Наблюдаемость** | — | WON'T HAVE | Логирование и мониторинг будут добавлены во 2–3 версиях при росте нагрузки. На старте можно обойтись базовыми средствами хостинга. |
| **Ёмкость** | — | WON'T HAVE | При планируемом объёме данных (до 500 событий в месяц) требования к хранилищу не критичны. Будут пересмотрены при масштабировании. |
| **Поддерживаемость** | — | WON'T HAVE | При планируемом объёме данных (до 500 событий в месяц) требования к хранилищу не критичны. Будут пересмотрены при масштабировании. |
| **Масштабируемость** | — | WON'T HAVE | Для MVP с нагрузкой до 100–200 пользователей не требуется. Будет добавлена в следующих версиях. |
| **Нормативно-правовые требования** | Передача персональных данных (почта, геолокация) между клиентом и сервером должна осуществляться по защищённому каналу | MUST HAVE | Передача персональных данных между клиентом и сервером осуществляется по защищённому каналу (152-ФЗ) |
